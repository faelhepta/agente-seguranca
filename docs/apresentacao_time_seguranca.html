Contexto

Por que esse processo existe?

⚠

O Problema

Vulnerabilidades chegando a producao sem revisao. Corrigir em producao custa 10x mais do que corrigir no desenvolvimento.

◉

A Situacao

A fabrica de software entrega codigo, documentacao e infraestrutura. Sem um gate de seguranca, qualquer falha vai direto para o ar.

✅

A Solucao

Um processo padrao de analise com checklist, agente de IA e parecer formal antes de cada deploy para producao.

Nossa missao: garantir que nenhum sistema com falha de segurança critica ou alta chegue ao ambiente de producao. Somos o ultimo filtro antes do usuario final.

1
Fase 1

Analise de Codigo Fonte

O que o agente vai verificar no codigo:

  • Autenticacao e autorizacao (JWT, OAuth, RBAC)
  • Injecoes — SQL, Command, LDAP, SSTI
  • Controle de acesso — IDOR, rotas sem auth
  • Criptografia — algoritmos, chaves, TLS
  • Credenciais e secrets hardcoded
  • Dependencias com CVEs conhecidos
  • Seguranca de API — CORS, rate limiting, headers
  • Logs com dados sensiveis expostos
  • Configuracoes de ambiente inseguras

Como rodar no agente:

# 1. Va para a pasta de codigo
cd projetos/PROJETO/codigo/

# 2. Inicie o agente
npx claude-code-templates@latest
    --agent security-engineer

# 3. Cole o prompt de analise
"Analise o codigo seguindo
 OWASP Top 10..."

# 4. Salve o resultado
../relatorios/01_codigo.md

5 prompts especializados disponíveis no Runbook para esta fase: analise geral, auth, dependencias, secrets e API.

2
Fase 2

Analise de Documentacao

Arquitetura e DFD

Fronteiras de confianca, dados sensiveis mapeados, fluxo de autenticacao, integracoes com terceiros e pontos de armazenamento criticos.

Conformidade LGPD

Quais dados pessoais sao coletados, base legal definida, politica de retencao, direitos dos titulares, DPA com terceiros e notificacao de incidentes.

Requisitos de Seguranca

Se os requisitos existem e estao sendo atendidos, classificacao de dados realizada, politica de senha documentada e processo de recuperacao seguro.

Atencao: a documentacao revela o que o codigo nao mostra — as decisoes de design, os fluxos de negocio e os dados que trafegam pelo sistema. Um sistema bem codificado com arquitetura mal projetada ainda e vulneravel.

3
Fase 3

Analise de Infraestrutura

Rede e Exposicao

Segmentacao DMZ/dados/app, banco de dados exposto?, WAF presente, portas de admin abertas para internet, TLS em toda comunicacao.

IAM e Acesso

Principio do menor privilegio nas roles, MFA exigido para admin, service accounts com permissoes minimas, rotacao de chaves definida.

Monitoramento

Logs de infra habilitados, alertas de seguranca configurados, retencao de logs definida (minimo 90 dias), SIEM previsto.

IaC (Terraform / CloudFormation)

Security groups abertos (0.0.0.0/0), buckets S3 publicos, discos sem criptografia, secrets nos arquivos IaC, AMIs desatualizadas.

Containers e Kubernetes

Containers rodando como root, imagens vulneraveis, secrets em variaveis de ambiente, privilegios excessivos, registry sem scan.

A Ferramenta

O Agente de Seguranca com IA

Usamos o Claude Code Security Engineer, um agente de IA especializado em seguranca, para escalar e padronizar as analises do time.

O que ele faz

Lê o codigo, documentos e diagramas e identifica vulnerabilidades com base em OWASP, CIS, NIST e STRIDE

O que ele nao faz

Substituir o julgamento humano. O analista valida, contextualizae emite o parecer final

Linha Vermelha

Criterios de Bloqueio de Deploy

Se qualquer um dos itens abaixo for identificado, o deploy e bloqueado automaticamente:

  • 🔐 Credenciais, chaves ou senhas hardcoded no codigo ou repositorio
  • 👈 SQL Injection ou Command Injection confirmado e exploravel
  • 🚫 Autenticacao bypassavel — acesso sem credenciais validas
  • 🔒 Dados sensiveis (PII, financeiros) expostos sem criptografia
  • 🔌 Banco de dados exposto diretamente para a internet
  • 🛠 CVE critico em dependencia core da aplicacao sem mitigacao
  • ⚠ Ausencia total de controle de acesso em endpoints sensiveis

Protocolo: Achado critico identificado → notificar coordenador imediatamente → aguardar orientacao antes de prosseguir com o relatorio.

Resultado Final

Os 3 Tipos de Parecer

✅

APROVADO

Nenhum achado critico ou alto identificado. O sistema pode ir para producao sem restricoes.

⚠

APROVADO COM RESSALVAS

Apenas achados medios e baixos. Deploy e permitido, desde que a fabrica assine um plano de correcao com prazos definidos.

❌

REPROVADO

Achados criticos ou altos presentes. Deploy bloqueado ate que as correcoes sejam implementadas e o sistema passe por nova analise do time de seguranca.

↻

Parecer REPROVADO ou APROVADO COM RESSALVAS com prazo vencido disparam o Processo de Reanalise — skill /reanalise-fabrica. O ciclo se repete ate o parecer APROVADO.

Times e Papeis

Responsabilidades

Analista de Seguranca

  • Organizar pastas e artefatos do projeto
  • Executar o agente em cada fase
  • Salvar e revisar os relatorios gerados
  • Preencher o checklist padrao
  • Escalar achados criticos imediatamente

Coordenador de Seguranca

  • Revisar e assinar o parecer final
  • Decidir em casos de divergencia
  • Comunicar resultado para a fabrica
  • Manter o checklist atualizado
  • Acompanhar planos de correcao

Fabrica de Software

  • Entregar os 3 artefatos obrigatoriamente
  • Corrigir achados no prazo acordado
  • Solicitar reanalise apos correcoes
  • Nao fazer deploy sem parecer aprovado
  • Tirar duvidas antes de codar
📢

Regra de ouro: Nenhum sistema vai para producao sem parecer do time de seguranca. Esta regra nao tem excecoes — nem para prazos, nem para urgencias.

Processo de Reanalise

Quando a reanalise e obrigatoria:

  • Parecer original: REPROVADO
  • Parecer: APROVADO COM RESSALVAS + 30 dias vencidos
  • Fabrica alega ter corrigido e solicita nova avaliacao

O que a fabrica DEVE entregar (sem isso nao inicia):

  • Codigo fonte atualizado
  • Lista de commits / Pull Requests das correcoes
  • Documento de evidencias por achado
  • Evidencias de testes realizados
  • Declaracao formal do responsavel tecnico
